# TEMA 1. Clases y objetos

## 1. ¿Cuáles son las cuatro características básicas de la programación orientada a objetos? Describe brevemente cada una

Las cuatro características básicas de la programación orientada a objetos son la encapsulación, la abstracción, la herencia y el polimorfismo. Estas permiten organizar el código de forma más estructurada y cercana a cómo se modelan los problemas en el mundo real.

La encapsulación consiste en agrupar datos y funciones en una misma unidad (la clase), controlando además el acceso a esos datos. La abstracción permite representar solo los aspectos relevantes de un objeto, ocultando los detalles innecesarios. Por su parte, la herencia permite crear nuevas clases a partir de otras existentes, reutilizando código. Finalmente, el polimorfismo permite que un mismo método tenga distintos comportamientos según el contexto.


## 2. Cita cuatro lenguajes populares que permitan la programación orientada a objetos

Existen numerosos lenguajes que soportan programación orientada a objetos. Algunos de los más populares son Java, C++, Python y C#. Todos ellos permiten definir clases, crear objetos y aplicar los principios básicos de la POO.

En el caso de C++, se trata de un lenguaje multiparadigma que permite tanto programación estructurada como orientada a objetos. Java, en cambio, está diseñado principalmente bajo el paradigma orientado a objetos. Python y C# también soportan ampliamente este modelo, aunque con diferencias en sintaxis y flexibilidad.


## 3. Los paradigmas anteriores a la POO, ¿Qué es la **programación estructurada**? y, todavía mejor, ¿Qué es la **programación modular**?

La programación estructurada es un paradigma basado en el uso de estructuras de control como secuencias, condicionales y bucles. Su objetivo es mejorar la claridad del código evitando el uso de saltos incontrolados (como goto) y organizando el flujo del programa de forma lógica.

La programación modular va un paso más allá y consiste en dividir un programa en módulos o funciones independientes. Cada módulo realiza una tarea específica y puede reutilizarse en diferentes partes del programa. Esto facilita el mantenimiento, la reutilización y la comprensión del código.

## 4. ¿Qué tres elementos definen a un objeto en programación orientada a objetos?

Un objeto en programación orientada a objetos se define por tres elementos: estado, comportamiento e identidad. Estos conceptos permiten diferenciar unos objetos de otros y describir su funcionamiento.

El estado está representado por los atributos o variables del objeto. El comportamiento corresponde a los métodos o funciones que puede ejecutar. La identidad es lo que permite distinguir un objeto de otro, incluso si tienen los mismos valores en sus atributos.

## 5. ¿Qué es una clase? ¿Es lo mismo que un objeto? ¿Qué es una instancia? ¿Todos los lenguajes orientados a objetos manejan el concepto de clase?

Una clase es una plantilla o modelo que define las propiedades (atributos) y comportamientos (métodos) de un tipo de objeto. No contiene datos concretos, sino la estructura que tendrán los objetos que se creen a partir de ella.

Un objeto no es lo mismo que una clase, sino una instancia de esta. Es decir, una instancia es un objeto concreto creado a partir de una clase. No todos los lenguajes orientados a objetos utilizan clases de la misma forma; algunos, como Java, son completamente basados en clases, mientras que otros como JavaScript utilizan modelos basados en prototipos.


## 6. ¿Dónde se almacenan en memoria los objetos? ¿Es igual en todos los lenguajes? ¿Qué es la **recolección de basura**? 

En Java, los objetos se almacenan normalmente en el heap (memoria dinámica), mientras que las variables que contienen referencias a esos objetos pueden almacenarse en la pila (stack). Este modelo facilita la gestión automática de memoria.

No todos los lenguajes funcionan igual. En C o C++, el programador controla manualmente la memoria, mientras que en Java esta gestión es automática. La recolección de basura es un mecanismo que libera automáticamente la memoria ocupada por objetos que ya no se utilizan, evitando fugas de memoria.


## 7. ¿Qué es un método? ¿Qué es la **sobrecarga de métodos**? 

Un método es una función definida dentro de una clase que describe el comportamiento de los objetos de dicha clase. Es equivalente a una función en C, pero asociada a un tipo concreto de dato.

La sobrecarga de métodos consiste en definir varios métodos con el mismo nombre pero con diferentes parámetros (tipo o número). Esto permite reutilizar el mismo nombre para operaciones similares, mejorando la legibilidad del código.

# TEMA 2. Encapsulación

## 1. En Programación Orientada a Objetos (POO), ¿Qué buscan la **encapsulación** y **la ocultación** de información? Enumera brevemente algunas ventajas de la ocultación de información.

La encapsulación busca agrupar datos y métodos dentro de una clase y controlar cómo se accede a ellos. La ocultación de información consiste en ocultar los detalles internos de la clase y mostrar solo lo necesario.

La principal ventaja es que se reduce la complejidad, se evita el uso incorrecto de los datos y se mejora el mantenimiento del código. También permite cambiar la implementación interna sin afectar al resto del programa.


## 2. ¿Qué se entiende por la **interfaz pública** de un objeto o clase en POO? Describe brevemente cómo se relaciona con la ocultación de información.

La interfaz pública de una clase es el conjunto de métodos y atributos accesibles desde fuera de la clase.

Está formada normalmente por métodos public. La ocultación de información consiste en mantener privados los detalles internos (private) y exponer solo lo necesario mediante la interfaz pública.


## 3. Brevemente: ¿Por qué hay que ser conscientes y diseñar con cuidado la **interfaz pública** de una clase? ¿Es fácil cambiarla?

La interfaz pública es muy importante porque define cómo se usa la clase desde el exterior. Si está mal diseñada, la clase será difícil de usar o poco flexible.

No es fácil cambiarla, ya que otros programas pueden depender de ella. Por eso debe diseñarse con cuidado desde el principio.

## 4. ¿Qué son las **invariantes de clase** y por qué la ocultación de información nos ayuda?

Una invariante es una regla que siempre debe cumplirse en el objeto (ej: edad ≥ 0).

La encapsulación ayuda porque evita modificar datos sin control.

## 5. Pon un ejemplo de una clase `Punto` en `Java`, con dos coordenadas, `x` e `y`, de tipo `double`, con un método `calcularDistanciaAOrigen`, y que haga uso de la ocultación de información. ¿Cuál es la interfaz pública de la clase `Punto`? ¿Qué significa `public` y `private`?

class Punto {
    private double x;
    private double y;

    public Punto(double x, double y) {
        this.x = x;
        this.y = y;
    }

    public double calcularDistanciaAOrigen() {
        return Math.sqrt(x*x + y*y);
    }
}

La interfaz pública está formada por el constructor y el método calcularDistanciaAOrigen.

public significa accesible desde fuera de la clase. private significa que solo se puede usar dentro de la propia clase.

## 6. En Java, ¿A quiénes se pueden aplicar los modificadores `public` o `private`?

En Java, public y private se pueden aplicar a atributos, métodos y constructores.

Sirven para controlar la visibilidad de los miembros de una clase.


## 7. En POO, la visibilidad puede ser pública o privada, pero ¿existen más tipos de visibilidad? ¿Qué ocurre en Java? ¿Y en otros lenguajes?

Además de public y private, existe la visibilidad por defecto (package-private) y protected.

En Java se usan estos cuatro niveles. Otros lenguajes pueden tener variantes, pero el concepto general es el mismo: controlar el acceso.


## 8. Responde: Los miembros de instancia privados de un objeto están ocultos para (a) otras clases o (b) otras instancias, aunque sean de la misma clase. Pon un ejemplo añadiendo un método `calcularDistanciaAPunto(Punto otro)` y explica la respuesta.

Los atributos privados no son accesibles desde otras clases, pero sí desde otras instancias de la misma clase.

class Punto {
    private double x, y;

    public double calcularDistanciaAPunto(Punto otro) {
        double dx = this.x - otro.x;
        double dy = this.y - otro.y;
        return Math.sqrt(dx*dx + dy*dy);
    }
}

Esto es posible porque una instancia de la misma clase puede acceder a los miembros privados de otra instancia de esa clase.


## 9. ¿Qué son los métodos "getter" y "setter" en los lenguajes orientados a objetos?

Los getters son métodos que devuelven el valor de un atributo privado.

Los setters son métodos que modifican el valor de un atributo privado de forma controlada.


## 10. Cuando nos referimos a que la ocultación de información mejora la "seguridad" del programa, ¿nos referimos a que no pueda ser "hackeado"?

La “seguridad” no significa protección contra hackers.

Significa evitar errores de programación, como modificar datos de forma incorrecta o romper reglas internas del objeto.

## 11. ¿Qué diferencia hay entre **miembro de instancia** y **miembro de clase**? ¿Los miembros de clase también se pueden ocultar?
Un miembro de instancia pertenece a cada objeto individual. Un miembro de clase (static) pertenece a la clase en general.

Los miembros de clase también pueden ser private o public, igual que los de instancia.


## 12. Brevemente: ¿Tiene sentido que los constructores sean privados?
Sí, puede tener sentido. Se usan para controlar la creación de objetos.

## 13. ¿Cómo se indican los **miembros de clase** en Java? Pon un ejemplo, en la clase `Punto` definida anteriormente, para que incluya miembros de clase que permitan saber cuáles son los valores `x` e `y` máximos que se han establecido en todos los puntos que se hayan creado hasta el momento.

Los miembros de clase se indican con static.

class Punto {
    private double x, y;

    private static double maxX = Double.NEGATIVE_INFINITY;
    private static double maxY = Double.NEGATIVE_INFINITY;

    public Punto(double x, double y) {
        this.x = x;
        this.y = y;

        if (x > maxX) maxX = x;
        if (y > maxY) maxY = y;
    }
}

# TEMA 3. Excepciones

## 1. Empecemos un tema sobre control de errores en lenguajes de programación, con algo básico. En C, donde no existen las excepciones, pongamos un ejemplo de una raíz que toma número flotante positivo. Queremos controlar el error si la función recibe un número negativo. El usuario debe ser informado pero desde fuera de la función `raiz` ¿Cómo indicamos ese error?. Enumera dos opciones diferentes de diseñar, poniendo un ejemplo de código de cada una.

En C no existen excepciones, por lo que el error se controla con valores de retorno o variables de estado.
double raiz(double x) {
    if (x < 0) return -1;
    return sqrt(x);
}

El llamador comprueba el valor devuelto.

double raiz(double x, int *error) {
    if (x < 0) {
        *error = 1;
        return 0;
    }
    *error = 0;
    return sqrt(x);
}

El error se comunica fuera de la función mediante un parámetro.


## 2. Brevemente ¿Qué es una **"excepción"**? ¿Con qué objetivo las usa un programador cuando implementa funciones o cuando las llama?

Una excepción es un mecanismo para indicar errores en tiempo de ejecución.

Permite separar el código normal del código de error. Se usa para propagar errores automáticamente hasta que alguien los controle.


## 3. Reescribe el mismo ejemplo de raiz, pero en Java, metiendo ese método en una clase `Calculadora` y llama a dicho método desde el método `main`, mostrando cómo se puede controlar desde fuera.

class Calculadora {
    static double raiz(double x) {
        if (x < 0) throw new IllegalArgumentException("Negativo");
        return Math.sqrt(x);
    }
}

public class Main {
    public static void main(String[] args) {
        try {
            System.out.println(Calculadora.raiz(-5));
        } catch (IllegalArgumentException e) {
            System.out.println("Error: " + e.getMessage());
        }
    }
}


## 4. ¿Qué es **"lanzar"** una excepción? ¿Qué es **"controlar"** o **"capturar"** una excepción? ¿Qué es que se **"propague"** una excepción? ¿Qué le va ocurriendo a las funciones en la pila de llamadas por donde se va propagando la excepción? ¿Las funciones que no la controlan se reanudan después de alguna forma? Explica con el mismo ejemplo anterior en Java de la raíz cuadrada.

Lanzar (throw) = crear una excepción cuando ocurre un error.
Capturar (catch) = manejar la excepción cuando ocurre.
Propagar = dejar que la excepción suba por la pila de llamadas.

class Calculadora {
    static double raiz(double x) {
        if (x < 0) throw new IllegalArgumentException();
        return Math.sqrt(x);
    }
}

Si no se captura, la excepción sube por las funciones llamadas hasta main.

Las funciones intermedias que no la capturan no continúan después del error, se aborta su ejecución.

## 5. ¿Qué ventajas tiene frente a C, la **"propagación natural"** de las excepciones a través de la pila (*stack*) de llamadas?

# Tema 4.1. Composición


## 1. En C, podemos crear estructuras mayores **componiendo** unas con otras, que suelen describirse como "A tiene-un/tiene-varios B". Pon un ejemplo, empleando `struct`, de una línea de puntos, donde puntos tienen dos coordenadas (`x` e `y`), y la línea esta hecha de dos puntos. Incluye una función para calcular la distancia entre puntos y otra para hallar la longitud de una línea.

En C, la composición puede realizarse incluyendo una estructura dentro de otra. De esta manera, una estructura más compleja puede construirse a partir de estructuras más simples. En este ejemplo, una línea estará formada por dos puntos, y cada punto tendrá dos coordenadas: x e y.

La relación que se modela es de tipo “Linea tiene dos Punto”. Esto permite organizar mejor la información y reutilizar estructuras ya definidas. Además, las funciones pueden trabajar directamente con estas estructuras para realizar cálculos geométricos, como la distancia entre dos puntos o la longitud de una línea.


#include <stdio.h>
#include <math.h>

struct Punto {
    double x;
    double y;
};

struct Linea {
    struct Punto inicio;
    struct Punto fin;
};

double distancia(struct Punto a, struct Punto b) {
    double dx = b.x - a.x;
    double dy = b.y - a.y;

    return sqrt(dx * dx + dy * dy);
}

double longitud(struct Linea l) {
    return distancia(l.inicio, l.fin);
}

int main() {
    struct Punto p1 = {0, 0};
    struct Punto p2 = {3, 4};

    struct Linea linea = {p1, p2};

    printf("Longitud: %.2f\n", longitud(linea));

    return 0;
}

En este caso, la función distancia calcula la separación entre dos puntos usando el teorema de Pitágoras. Después, la función longitud reutiliza esa función para calcular la longitud de una línea completa a partir de sus puntos inicial y final.



## 2. Ahora transforma ese ejemplo a orientación a objetos con Java, para tener un primer ejemplo de **composición** en orientación a objetos. Crea una clase `Punto`, y una clase `Linea`. La clase `Punto` debe tener un método para calcular distancia a otro `Punto` y `Linea` debe tener un método para calcular su longitud. Gracias a la ocultación de información, supera a C, garantizando que los puntos sean inmutables, al igual que la línea, que una vez creada, no queremos que se modifique de qué a qué puntos va dicha línea.  

En Java, la composición se realiza haciendo que una clase tenga objetos de otra clase como atributos. En este ejemplo, la clase Linea estará compuesta por dos objetos de tipo Punto. A diferencia de C, Java permite aplicar ocultación de información mediante atributos privados y controlar cómo se accede a ellos.

Para conseguir inmutabilidad, los atributos se declaran como private final, y no se crean métodos modificadores (setters). Así, una vez creado un objeto Punto o Linea, sus valores no podrán cambiarse. Esto aumenta la seguridad y evita modificaciones accidentales del estado de los objetos.

public class Punto {
    private final double x;
    private final double y;

    public Punto(double x, double y) {
        this.x = x;
        this.y = y;
    }

    public double getX() {
        return x;
    }

    public double getY() {
        return y;
    }

    public double distancia(Punto otro) {
        double dx = otro.x - this.x;
        double dy = otro.y - this.y;

        return Math.sqrt(dx * dx + dy * dy);
    }
}

public class Linea {
    private final Punto inicio;
    private final Punto fin;

    public Linea(Punto inicio, Punto fin) {
        this.inicio = inicio;
        this.fin = fin;
    }

    public Punto getInicio() {
        return inicio;
    }

    public Punto getFin() {
        return fin;
    }

    public double longitud() {
        return inicio.distancia(fin);
    }
}

La principal ventaja respecto a C es que el estado interno queda protegido. No es posible modificar directamente las coordenadas de un punto ni cambiar los extremos de una línea después de crearla. Además, cada clase se responsabiliza de sus propias operaciones, siguiendo el principio de encapsulación propio de la orientación a objetos.

## 3. ¿Qué significa la **multiplicidad** en la composición? En el ejemplo anterior, ¿cuál es la multiplicidad entre `Linea` y `Punto`? Indícalo expresando la multiplicidad en ambas direcciones, de `Linea` a `Punto` y de `Punto` a `Linea`.

La multiplicidad indica cuántos objetos de una clase pueden relacionarse con objetos de otra clase dentro de una asociación o composición. Se utiliza en diseño orientado a objetos para describir relaciones entre clases de forma más precisa.

En el ejemplo anterior, una Linea está formada exactamente por dos objetos Punto: uno de inicio y otro de fin. Por tanto, la multiplicidad de Linea hacia Punto es 2, ya que cada línea necesita exactamente dos puntos para existir.

En sentido contrario, un Punto puede pertenecer a ninguna, una o varias líneas diferentes. Un mismo punto podría reutilizarse como extremo de muchas líneas distintas. Por ello, la multiplicidad de Punto hacia Linea sería 0..*, que significa “cero o muchas”.

Estas multiplicidades ayudan a entender cómo se relacionan los objetos y qué restricciones existen entre ellos. Son especialmente útiles al diseñar diagramas UML y estructuras de clases más complejas.


## 4. ¿Qué significa composición **fuerte** y composición **débil**? ¿Qué consecuencia implica en relación al ciclo de vida de los objetos? Indica a cuál solemos referirnos como **"asociación o agregación"** y a cuál como **"composición"** propiamente.

La composición fuerte ocurre cuando un objeto propietario controla completamente la existencia de los objetos que contiene. Esto significa que los objetos internos no suelen existir de manera independiente y dependen del ciclo de vida del objeto principal.

Por ejemplo, si una clase Casa crea y destruye directamente sus objetos Habitacion, las habitaciones dejarán de existir cuando desaparezca la casa. A esta relación fuerte es a la que normalmente se denomina “composición” propiamente dicha.

La composición débil ocurre cuando los objetos relacionados pueden existir de forma independiente. El objeto principal simplemente mantiene una referencia hacia otros objetos, pero no controla completamente su creación ni su destrucción. A este tipo de relación suele llamársele “agregación” o “asociación”.

La diferencia principal está en el ciclo de vida de los objetos. En la composición fuerte, el objeto contenido depende del contenedor para existir. En la composición débil, ambos objetos pueden seguir existiendo por separado aunque la relación desaparezca.

## 5. Cuando una clase usa a otra al recibirla o devolverla como parámetro en algún método, al hacer `new` dentro de un método, o al usarlas como variables locales, ¿hablamos de composición o de **"dependencia"**?

Cuando una clase utiliza otra únicamente de manera temporal, por ejemplo como parámetro de un método, valor de retorno, variable local o creando objetos dentro de un método, se habla de dependencia y no de composición.

La composición implica que un objeto mantiene otro objeto como parte de su estado interno mediante atributos. En cambio, la dependencia representa un uso puntual o temporal entre clases, sin que exista una relación permanente almacenada dentro del objeto.

Por ejemplo, si una clase Calculadora recibe un objeto Punto en un método para realizar un cálculo, existe dependencia porque el objeto se usa momentáneamente. Sin embargo, si una clase Linea guarda dos objetos Punto como atributos, entonces sí existe composición.

Las dependencias son relaciones más débiles que la composición. Aun así, siguen siendo importantes porque indican qué clases necesitan conocer o utilizar a otras para funcionar correctamente.


## 6. En el ejemplo anterior de línea y punto, programa la relación entre `Linea` y `Punto` de dos formas. Una **como composición fuerte**, donde el ciclo de vida de los puntos está ligado al de Linea y otra **como composición débil**, donde no.

En una composición fuerte, los objetos internos son creados directamente por el objeto contenedor y dependen completamente de él. En este caso, Linea crea sus propios puntos y no comparte referencias con otros objetos externos. Así, los puntos forman parte exclusiva de la línea.

public class Punto {
    private final double x;
    private final double y;

    public Punto(double x, double y) {
        this.x = x;
        this.y = y;
    }

    public double distancia(Punto otro) {
        double dx = otro.x - this.x;
        double dy = otro.y - this.y;

        return Math.sqrt(dx * dx + dy * dy);
    }
}

public class Linea {
    private final Punto inicio;
    private final Punto fin;

    public Linea(double x1, double y1, double x2, double y2) {
        this.inicio = new Punto(x1, y1);
        this.fin = new Punto(x2, y2);
    }

    public double longitud() {
        return inicio.distancia(fin);
    }
}

En la composición débil, los puntos existen independientemente de la línea y pueden compartirse entre varios objetos. La línea simplemente recibe referencias a puntos ya creados desde el exterior.

public class Linea {
    private final Punto inicio;
    private final Punto fin;

    public Linea(Punto inicio, Punto fin) {
        this.inicio = inicio;
        this.fin = fin;
    }

    public double longitud() {
        return inicio.distancia(fin);
    }
}

La diferencia principal está en el control del ciclo de vida. En la composición fuerte, los puntos nacen junto con la línea. En la débil, los puntos pueden existir antes, seguir existiendo después o incluso pertenecer a varias líneas simultáneamente.


## 7. En Java, en la composición fuerte, ¿cuando el contenedor destruye los objetos? No se observa que `Linea` destruya los `Punto` explícitamente, ¿Por qué?

En Java, la destrucción de objetos no se realiza manualmente como ocurre en lenguajes como C++. En su lugar, existe un recolector de basura (Garbage Collector) que libera automáticamente la memoria de los objetos que ya no pueden utilizarse.

En el ejemplo de composición fuerte, los objetos Punto pertenecen únicamente a la Linea. Cuando el objeto Linea deja de tener referencias activas, también desaparecen las únicas referencias hacia sus puntos. En consecuencia, el recolector de basura detecta que esos objetos ya no son accesibles y libera su memoria automáticamente.

Por ello, no aparece ningún código explícito destruyendo los puntos. Java oculta la gestión manual de memoria para evitar errores frecuentes, como liberar memoria dos veces o acceder a memoria ya destruida.

La composición fuerte sigue existiendo conceptualmente, porque el ciclo de vida de los puntos depende de la línea. Sin embargo, la destrucción física de los objetos queda delegada al sistema automático de gestión de memoria de Java.


## 8. Pon un ejemplo de composicion débil entre un departamento que tiene varios profesores. Implementa dos composiciones a la vez: entre el departamento y todos sus profesores y entre el departamento y su director, que es un profesor del departamento. Siempre debe haber un director en el departamento desde el inicio. Lanza excepciones si se viola la invariante. Emplea arrays primitivos de Java, estilo `Profesor[]`, con máximo 50, pero no rompas la encapsulación, no desveles que estás empleando un array, permite añadir un `Profesor` al final de la lista, y eliminar un profesor dada su posición. Da acceso a los profesores con un método para saber cuántos hay y otro para obtener un profesor por posición. El director se puede cambiar por otro profesor del departamento. Sin embargo, ten en cuenta esta invariante de clase: el director debe formar siempre parte de la lista de profesores, es decir, ten cuidado al cambiar el director o al eliminar un profesor.

En este ejemplo existe una composición débil porque los objetos Profesor pueden existir independientemente del Departamento. El departamento únicamente mantiene referencias a ellos. Además, aparecen dos relaciones simultáneas: una relación con todos los profesores y otra específica con el profesor que actúa como director.

La clase debe mantener una invariante importante: siempre debe existir un director y este debe pertenecer a la lista de profesores del departamento. Para garantizarlo, se lanzan excepciones cuando alguna operación intenta dejar el objeto en un estado inválido.

public class Profesor {
    private String nombre;

    public Profesor(String nombre) {
        this.nombre = nombre;
    }

    public String getNombre() {
        return nombre;
    }
}

public class Departamento {
    private Profesor[] profesores;
    private int numProfesores;
    private Profesor director;

    public Departamento(Profesor director) {
        if (director == null) {
            throw new IllegalArgumentException("Debe existir un director");
        }

        profesores = new Profesor[50];
        profesores[0] = director;
        numProfesores = 1;
        this.director = director;
    }

    public void addProfesor(Profesor profesor) {
        if (numProfesores >= 50) {
            throw new IllegalStateException("Departamento lleno");
        }

        profesores[numProfesores] = profesor;
        numProfesores++;
    }

    public void removeProfesor(int pos) {
        if (pos < 0 || pos >= numProfesores) {
            throw new IndexOutOfBoundsException();
        }

        if (profesores[pos] == director) {
            throw new IllegalStateException(
                "No se puede eliminar al director"
            );
        }

        for (int i = pos; i < numProfesores - 1; i++) {
            profesores[i] = profesores[i + 1];
        }

        profesores[numProfesores - 1] = null;
        numProfesores--;
    }

    public int getNumProfesores() {
        return numProfesores;
    }

    public Profesor getProfesor(int pos) {
        if (pos < 0 || pos >= numProfesores) {
            throw new IndexOutOfBoundsException();
        }

        return profesores[pos];
    }

    public Profesor getDirector() {
        return director;
    }

    public void setDirector(Profesor nuevoDirector) {
        boolean encontrado = false;

        for (int i = 0; i < numProfesores; i++) {
            if (profesores[i] == nuevoDirector) {
                encontrado = true;
                break;
            }
        }

        if (!encontrado) {
            throw new IllegalArgumentException(
                "El director debe pertenecer al departamento"
            );
        }

        director = nuevoDirector;
    }
}

El uso de métodos como getNumProfesores() y getProfesor(int pos) evita exponer directamente el array interno. Así, la implementación puede modificarse en el futuro sin afectar al código que utiliza la clase, manteniendo la encapsulación.


## 9. En Java, existen también `List`, cambia y muestra cómo sería el código anterior empleando `List` en vez de arrays primitivos. ¿Qué parte del código original te has ahorrado? Además, fíjate en el método `getProfesor(int pos)`: si en su lugar existiera un método que devolviera todos los profesores a la vez, ¿qué problema tendría devolver directamente la lista interna? ¿Cómo lo resolverías?

La interfaz List simplifica mucho el manejo de colecciones en Java. Ya no es necesario gestionar manualmente el tamaño, desplazar elementos ni controlar posiciones libres dentro de un array. La implementación más habitual es ArrayList.

import java.util.ArrayList;
import java.util.Collections;
import java.util.List;

public class Departamento {
    private List<Profesor> profesores;
    private Profesor director;

    public Departamento(Profesor director) {
        if (director == null) {
            throw new IllegalArgumentException("Debe existir un director");
        }

        profesores = new ArrayList<>();
        profesores.add(director);

        this.director = director;
    }

    public void addProfesor(Profesor profesor) {
        profesores.add(profesor);
    }

    public void removeProfesor(int pos) {
        Profesor profesor = profesores.get(pos);

        if (profesor == director) {
            throw new IllegalStateException(
                "No se puede eliminar al director"
            );
        }

        profesores.remove(pos);
    }

    public int getNumProfesores() {
        return profesores.size();
    }

    public Profesor getProfesor(int pos) {
        return profesores.get(pos);
    }

    public Profesor getDirector() {
        return director;
    }

    public void setDirector(Profesor nuevoDirector) {
        if (!profesores.contains(nuevoDirector)) {
            throw new IllegalArgumentException(
                "El director debe pertenecer al departamento"
            );
        }

        director = nuevoDirector;
    }

    public List<Profesor> getProfesores() {
        return Collections.unmodifiableList(profesores);
    }
}

Gracias a List, desaparece gran parte del código dedicado a mover elementos, controlar el número de profesores o gestionar posiciones vacías del array. El código resulta más corto, legible y menos propenso a errores.

Si se devolviera directamente la lista interna mediante un getter, código externo podría modificarla libremente, eliminando profesores o añadiendo otros sin respetar las invariantes de la clase. Esto rompería la encapsulación.

Para evitarlo, puede devolverse una vista no modificable usando Collections.unmodifiableList(profesores). Así, el exterior puede consultar la lista pero no alterarla directamente.


## 10. Al igual que ocurre con las excepciones en Java, que pueden encerrar causas (que son excepciones), de forma recursiva, suponen un tipo especial de composiciones, denominadas composiciones recursivas. Pon un ejemplo en Java de una `Persona`, que sea inmutable, y que tiene una madre, que es otra `Persona`. Haz un main con un ejemplo de uso con una familia de personas, desde el nieto hasta la abuela. Enumera algún otro ejemplo clásico de composiciones recursivas.

Una composición recursiva ocurre cuando una clase contiene referencias a objetos de su mismo tipo. En este caso, una Persona tendrá una madre que también es otra Persona. Este tipo de estructura permite representar jerarquías o relaciones encadenadas.

El siguiente ejemplo muestra una clase inmutable. Una vez creada una persona, no puede modificarse ni su nombre ni su madre.

public class Persona {
    private final String nombre;
    private final Persona madre;

    public Persona(String nombre, Persona madre) {
        this.nombre = nombre;
        this.madre = madre;
    }

    public String getNombre() {
        return nombre;
    }

    public Persona getMadre() {
        return madre;
    }
}

public class Main {
    public static void main(String[] args) {
        Persona abuela = new Persona("Carmen", null);

        Persona madre = new Persona("Laura", abuela);

        Persona hijo = new Persona("Miguel", madre);

        System.out.println(hijo.getNombre());
        System.out.println(hijo.getMadre().getNombre());
        System.out.println(
            hijo.getMadre().getMadre().getNombre()
        );
    }
}

Otros ejemplos clásicos de composiciones recursivas son los directorios de un sistema de archivos, donde una carpeta contiene otras carpetas, los nodos de árboles binarios y las estructuras de excepciones encadenadas mediante causas (Throwable cause) en Java.



## 11. ¿Qué son las relaciones de composición "bidireccionales"? ¿Qué habría que hacer para implementar este tipo de relación en el ejemplo de `Profesor` y `Departamento`?

Una relación bidireccional ocurre cuando ambas clases conocen la existencia de la otra. No solo el objeto contenedor mantiene referencias hacia los objetos contenidos, sino que estos también guardan una referencia hacia el contenedor.

En el ejemplo anterior, el Departamento conoce a sus Profesor, pero los profesores no saben a qué departamento pertenecen. Para convertir la relación en bidireccional, habría que añadir un atributo Departamento departamento dentro de la clase Profesor.

public class Profesor {
    private String nombre;
    private Departamento departamento;

    public Profesor(String nombre) {
        this.nombre = nombre;
    }

    public Departamento getDepartamento() {
        return departamento;
    }

    public void setDepartamento(Departamento departamento) {
        this.departamento = departamento;
    }
}

Además, al añadir o eliminar profesores desde Departamento, habría que actualizar también la referencia inversa dentro de cada profesor. Esto obliga a mantener sincronizadas ambas partes de la relación para evitar inconsistencias.

Las relaciones bidireccionales facilitan navegar entre objetos en ambos sentidos, pero también aumentan la complejidad del diseño y el riesgo de errores si no se actualizan correctamente las referencias.



# Tema 4.2. Herencia

## 1. En orientación a objetos, ¿qué es la **herencia** y su relación con "A es-un B"?. Explica las dos implicaciones principales: (1) **compatibilidad de tipos** y (2) **herencia de estado y comportamiento**. Pon un ejemplo en Java muy sencillo, donde un `Soldado` tiene un `nombre` (privado) y un método `saludar()` que muestra su nombre. Hay dos subtipos: un `Artillero`, que es capaz de disparar cohetes y un `Zapador` que pone minas, ambos heredan el atributo nombre y la capacidad de saludar. Además, y de forma específica, el artillero tiene un número de cohetes y el zapador un número de minas, accesibles mediante "getters" específicos. Respecto a la compatibilidad de tipos, aprovechémosla: crea un array de `Soldado`, mete varios de distinto tipo (son todos compatibles con `Soldado`). Recórrela y que todos te saluden.

La herencia es un mecanismo de la orientación a objetos que permite crear una clase nueva a partir de otra ya existente. La relación que representa suele describirse como “A es-un B”. Por ejemplo, un Artillero es un Soldado y un Zapador también es un Soldado. Esto significa que ambos comparten características comunes propias de cualquier soldado.

La herencia tiene dos implicaciones principales. La primera es la compatibilidad de tipos: cualquier objeto de una subclase puede utilizarse donde se espere un objeto de la superclase. La segunda es la herencia de estado y comportamiento: las subclases reciben automáticamente atributos y métodos heredables de la clase base.

public class Soldado {
    private String nombre;

    public Soldado(String nombre) {
        this.nombre = nombre;
    }

    public String getNombre() {
        return nombre;
    }

    public void saludar() {
        System.out.println("Hola, soy " + nombre);
    }
}

public class Artillero extends Soldado {
    private int cohetes;

    public Artillero(String nombre, int cohetes) {
        super(nombre);
        this.cohetes = cohetes;
    }

    public int getCohetes() {
        return cohetes;
    }
}

public class Zapador extends Soldado {
    private int minas;

    public Zapador(String nombre, int minas) {
        super(nombre);
        this.minas = minas;
    }

    public int getMinas() {
        return minas;
    }
}

public class Main {
    public static void main(String[] args) {
        Soldado[] soldados = {
            new Artillero("Carlos", 5),
            new Zapador("Luis", 3),
            new Artillero("Ana", 8)
        };

        for (Soldado s : soldados) {
            s.saludar();
        }
    }
}

En el ejemplo, Artillero y Zapador heredan el atributo nombre y el método saludar(). Además, cada subtipo añade comportamiento y estado específico. Gracias a la compatibilidad de tipos, todos pueden almacenarse juntos en un array de Soldado.


## 2. Al crear los soldados concretos, ¿cuántos constructores se ejecutan y en qué orden? ¿Qué significa `super` dentro de un constructor? Si la clase base no tiene visible el constructor sin parámetros, ¿debo llamar a `super` siempre? 

Cuando se crea un objeto de una subclase, no solo se ejecuta el constructor de dicha subclase. También se ejecutan los constructores de todas las superclases implicadas en la herencia. Esto ocurre porque antes de construir la parte específica del subtipo, primero debe construirse la parte heredada.

Por ejemplo, al crear un Artillero, primero se ejecuta el constructor de Soldado y después el de Artillero. El orden siempre va desde la superclase hacia la subclase.

public Artillero(String nombre, int cohetes) {
    super(nombre);
    this.cohetes = cohetes;
}

La palabra super permite invocar explícitamente un constructor de la clase base. En este caso, super(nombre) llama al constructor de Soldado para inicializar el atributo nombre.

Si la superclase no tiene un constructor sin parámetros accesible, es obligatorio llamar manualmente a super(...) indicando qué constructor debe usarse. De lo contrario, el compilador producirá un error, ya que Java intenta automáticamente llamar a super() cuando no se especifica nada.



## 3. Respecto a los objetos de subclases en memoria, los atributos privados de la superclase, ¿forman parte de una instancia de la subclase en memoria? En caso afirmativo ¿implica que se puedan usar desde el código de la subclase? Explícalo con el ejemplo de `Soldado` y alguna de sus subclases.

Sí, los atributos privados de la superclase forman parte física del objeto de la subclase en memoria. Un objeto Artillero contiene tanto la información heredada de Soldado como la información específica del propio Artillero.

Sin embargo, que los atributos existan físicamente dentro del objeto no significa que puedan accederse directamente desde el código de la subclase. La palabra private sigue impidiendo el acceso directo desde clases hijas.

En el ejemplo, el atributo nombre existe dentro del objeto Artillero, pero no puede utilizarse directamente porque pertenece a la parte privada de Soldado. Para acceder a él es necesario usar métodos públicos o protegidos proporcionados por la superclase, como getNombre()



## 4. ¿Qué implica en términos de **extensibilidad** de código el hecho de que sean compatibles a nivel de tipos? Ilustra esto añadiendo un nuevo tipo de `Soldado` y demostrando que el código para pedir el saludo a todos los soldados no se modifica.

La compatibilidad de tipos permite extender programas sin modificar código ya existente. Esto es una ventaja importante de la herencia, porque pueden añadirse nuevos subtipos reutilizando estructuras generales definidas para la superclase.

Por ejemplo, puede añadirse un nuevo tipo Medico que también sea un Soldado. Como sigue siendo compatible con el tipo Soldado, podrá utilizarse automáticamente en cualquier estructura preparada para trabajar con soldados.

public class Medico extends Soldado {

}

public class Main {

Obsérvese que el código del recorrido no cambia en absoluto. El array sigue siendo de tipo Soldado[] y el bucle sigue llamando a saludar(). Esto hace que el software sea más extensible, ya que pueden añadirse nuevos tipos sin modificar las partes generales del programa.




## 5. En Java, cuando trabajo con referencias y herencia. ¿Puedo tener una referencia del supertipo que apunte a objetos reales de un subtipo? ¿Puedo invocar con la referencia del supertipo a métodos públicos del subtipo? ¿En qué consiste el **"upcasting"** y el **"downcasting"**? ¿Qué es el `instanceof`? Pon un ejemplo de recorrido de un array de `Soldado`, comprobando que, si el objeto real es un `Artillero`, solicite el número de cohetes que tiene y los imprima.

Sí, en Java una referencia de la superclase puede apuntar a objetos reales de cualquiera de sus subclases. Esto es precisamente lo que permite aprovechar la compatibilidad de tipos en la herencia.

Sin embargo, una referencia del supertipo solo puede invocar métodos visibles en dicho supertipo. Aunque el objeto real sea un Artillero, si la referencia es de tipo Soldado, no podrá llamarse directamente a getCohetes().

El proceso de tratar un objeto de subtipo como si fuese de un supertipo se llama upcasting y ocurre automáticamente. El proceso contrario, convertir una referencia general en una más específica, se llama downcasting y requiere un cast explícito. Para evitar errores, suele comprobarse previamente el tipo real mediante instanceof.

public class Main {
    public static void main(String[] args) {

        Soldado[] soldados = {
            new Artillero("Carlos", 5),
            new Zapador("Luis", 3),
            new Artillero("Ana", 8)
        };

        for (Soldado s : soldados) {

            s.saludar();

            if (s instanceof Artillero) {

                Artillero a = (Artillero) s;

                System.out.println(
                    "Cohetes: " + a.getCohetes()
                );
            }
        }
    }
}

En este ejemplo, el array contiene referencias de tipo Soldado, pero algunos objetos reales son Artillero. Gracias a instanceof, puede comprobarse el tipo real antes de realizar el downcasting y acceder así a métodos específicos del subtipo.


## 6. Respecto a la ocultación de información y herencia, ¿qué significa acceso **"protegido"** de métodos y/o atributos? ¿Cómo se implementa en Java? Pon un ejemplo de uso de en la clase `Soldado` para que su nombre sea protegido y pueda usarse en el método de poner bombas del `Zapador`.

El acceso protegido (protected) en orientación a objetos es un nivel de visibilidad intermedio entre private y public. Permite que los atributos y métodos sean accesibles desde la propia clase, desde sus subclases y también desde clases del mismo paquete. Esto resulta especialmente útil en herencia, ya que permite reutilizar estado y comportamiento sin exponer completamente los datos al exterior.

En Java, el modificador protected se utiliza delante del atributo o método. A diferencia de private, las subclases pueden acceder directamente a los miembros protegidos de la superclase, lo que facilita la extensión del comportamiento sin necesidad de getters en todos los casos.

public class Soldado {
    protected String nombre;

    public Soldado(String nombre) {
        this.nombre = nombre;
    }
}

public class Zapador extends Soldado {

    public Zapador(String nombre) {
        super(nombre);
    }

    public void ponerMinas() {
        System.out.println(nombre + " está colocando minas");
    }
}

En este ejemplo, el atributo nombre es accesible directamente desde Zapador gracias a protected. Esto permite reutilizar el estado heredado sin romper la encapsulación completamente, aunque sigue siendo más expuesto que private.


## 7. En los lenguajes orientados a objetos ¿hay una **clase base** para todos los objetos? ¿Ocurre en todos los lenguajes? ¿Qué ocurre en Java?

En muchos lenguajes orientados a objetos existe una clase raíz común de la que heredan todos los demás tipos. Esta idea permite tratar cualquier objeto de forma genérica y definir comportamientos básicos compartidos, como comparación o representación en texto.

Sin embargo, esto no ocurre en todos los lenguajes. En algunos sistemas de herencia más primitivos o basados en prototipos, como JavaScript, no existe una única clase base explícita, sino otro mecanismo de delegación.

En Java sí existe una clase base universal llamada Object. Todas las clases creadas heredan directa o indirectamente de Object, incluso si no se indica explícitamente. Esto permite que cualquier objeto pueda ser tratado de forma genérica y que disponga de métodos comunes como toString(), equals() o hashCode().


## 8. ¿Qué es la **"herencia múltiple"**? ¿Existe en Java herencia múltiple?

La herencia múltiple es la capacidad de una clase para heredar de más de una clase base al mismo tiempo. Esto permitiría combinar comportamiento y estado de varias clases superiores en una sola subclase.

Este mecanismo puede generar problemas de ambigüedad, como el conflicto de qué implementación usar si dos superclases definen el mismo método. Por este motivo, no todos los lenguajes lo permiten de forma directa.

En Java no existe herencia múltiple de clases. Es decir, una clase solo puede extender una única clase padre. Sin embargo, Java permite una forma limitada de herencia múltiple a través de interfaces, lo que evita muchos de los problemas clásicos de ambigüedad.


## 9. Las excepciones en los lenguajes orientados a objetos son objetos. Por tanto, se pueden crear excepciones personalizadas. Pon un ejemplo en Java de una excepción personalizada (`UsuarioNoEncontradoException`), que sea *no controlada* y que además este compuesto con un `Usuario`, para saber qué `Usuario` dio el problema. Permite además que se pueda incluir la causa, es decir, sobrecarga el constructor para tener una versión que permita añadir la causa subyacente. 

Las excepciones personalizadas permiten representar errores específicos del dominio de la aplicación. Al ser objetos, pueden contener información adicional que ayude a diagnosticar el problema, como el usuario implicado o la causa original del error.

Una excepción no controlada en Java se crea extendiendo RuntimeException. Esto evita la obligación de declararla o capturarla explícitamente, aunque sigue pudiendo manejarse si se desea.

public class Usuario {
    private String nombre;

    public Usuario(String nombre) {
        this.nombre = nombre;
    }

    public String getNombre() {
        return nombre;
    }
}

public class UsuarioNoEncontradoException extends RuntimeException {

    private final Usuario usuario;

    public UsuarioNoEncontradoException(Usuario usuario) {
        super("Usuario no encontrado: " + usuario.getNombre());
        this.usuario = usuario;
    }

    public UsuarioNoEncontradoException(Usuario usuario, Throwable causa) {
        super("Usuario no encontrado: " + usuario.getNombre(), causa);
        this.usuario = usuario;
    }

    public Usuario getUsuario() {
        return usuario;
    }
}

En este ejemplo, la excepción incluye el objeto Usuario que provocó el error, y también permite almacenar una causa subyacente mediante sobrecarga del constructor, facilitando el rastreo de errores en sistemas más complejos.


## 10. Herencia vs. Composición. Se dice que no se debe emplear herencia simplemente por reutilizar código, es decir, que si quiero reutilizar código simplemente, no debo pensar en herencia como primera opción ¿por qué?

La herencia no debe usarse únicamente como mecanismo de reutilización de código porque implica una relación semántica fuerte del tipo “es-un”. Si esta relación no es real, el diseño se vuelve incorrecto y difícil de mantener, ya que las subclases quedan acopladas a la estructura de la superclase.

Cuando se usa herencia solo para reutilizar código, se corre el riesgo de forzar jerarquías artificiales. Esto puede provocar que futuras modificaciones en la clase base afecten a clases que no deberían verse impactadas, rompiendo el principio de diseño abierto/cerrado.

Por este motivo, en muchos casos es preferible utilizar composición, donde una clase contiene a otra y reutiliza su funcionalidad sin heredar su jerarquía. Esto reduce el acoplamiento y hace el diseño más flexible y fácil de evolucionar.


## 11. Herencia vs. Composición. Se dice que se debe *"favorecer la composición frente a la herencia"*, ¿por qué?

Se favorece la composición frente a la herencia porque la composición proporciona mayor flexibilidad en el diseño de software. En lugar de depender de una jerarquía rígida, una clase puede combinar diferentes comportamientos simplemente utilizando objetos internos.

La herencia introduce un acoplamiento fuerte entre clases, ya que las subclases dependen directamente de la implementación de la superclase. Esto puede dificultar cambios futuros, porque modificar la clase base puede afectar a todas sus subclases de forma inesperada.

En cambio, con composición, las relaciones son más flexibles y reemplazables en tiempo de diseño. Esto permite cambiar comportamientos sin modificar jerarquías completas, lo que mejora la mantenibilidad, reutilización y extensibilidad del sistema.


## 12. Herencia vs. Composición. Se dice que la *"herencia rompe la encapsulación"*, ¿a qué se refiere esto?

Se dice que la herencia puede romper la encapsulación porque una subclase queda dependiente de los detalles internos de implementación de la superclase. Aunque los atributos estén declarados como private, el comportamiento heredado sigue condicionado por cómo está implementada la clase base, lo que crea un acoplamiento fuerte entre ambas.

Además, la subclase puede verse afectada por cambios en la superclase sin que exista una interfaz explícita que lo controle. Por ejemplo, si se modifica un método heredado o se añade lógica interna, las subclases pueden cambiar su comportamiento de forma inesperada, incluso sin modificar su código.

Esto provoca que la superclase no esté completamente “encapsulada” como unidad independiente, ya que su diseño interno influye directamente en el comportamiento de las clases hijas. Por eso se dice que la herencia expone más de lo deseable en comparación con la composición, donde las dependencias están más controladas.



## 13. Pongamos un ejemplo de dos alternativas para lo mismo. Tenemos un `Estudiante` y un `Trabajador`, ambos tienen datos en común: el DNI y el nombre. Modelemos esto de dos formas: uno por herencia, con una superclase `Persona`, y otro con composición, con una clase `DatosPersonales`. Se debe recibir una instancia de `DatosPersonales` en el constructor de la clase `Estudiante` y `Trabajador`.

En el modelo basado en herencia, se utiliza una superclase Persona que contiene los datos comunes (dni y nombre). Tanto Estudiante como Trabajador heredan directamente de ella, reutilizando esos atributos y comportamiento común. Este enfoque representa una relación “es-un”.

public class Persona {

    private String nombre;

        this.dni = dni;
        this.nombre = nombre;

    }

    public String getNombre() {
        return nombre;
    }
}

public class Estudiante extends Persona {
    public Estudiante(String dni, String nombre) {

}

En el modelo basado en composición, los datos comunes se encapsulan en una clase independiente llamada DatosPersonales. Tanto Estudiante como Trabajador reciben una instancia de esta clase en su constructor y la almacenan como atributo. Esto representa una relación “tiene-un”.


public class DatosPersonales {

# Tema 5. Polimorfismo

## 1. Brevemente, ¿qué es el **"polimorfismo"** y para qué sirve en programación orientada a objetos? ¿qué es la **"sobreescritura"** de métodos?

El polimorfismo es un mecanismo de la programación orientada a objetos que permite que una misma referencia de tipo general pueda comportarse de diferentes maneras según el objeto real al que apunte. En otras palabras, permite tratar objetos de distintas subclases como si fueran de su clase base, pero ejecutando su comportamiento específico.

Su utilidad principal es la extensibilidad y la flexibilidad del código, ya que permite escribir algoritmos genéricos que funcionan con una superclase sin necesidad de conocer todos los tipos concretos derivados. Esto reduce la necesidad de estructuras condicionales basadas en tipos.

La sobreescritura de métodos (overriding) ocurre cuando una subclase redefine un método heredado de la superclase, proporcionando una implementación específica. De este modo, el mismo método puede tener comportamientos distintos dependiendo del tipo real del objeto.


## 2. ¿En qué consiste la **"ligadura dinámica"** o **"enlace tardío"**? ¿qué relación tiene con el polimorfismo? ¿hay que indicarlos explícitamente al programar o depende esto del lenguaje? Compara C++ y Java. Indicalo después también para Python.

La ligadura dinámica (o enlace tardío) consiste en que la decisión de qué método ejecutar no se toma en tiempo de compilación, sino en tiempo de ejecución. Es decir, el sistema determina en ejecución qué implementación concreta de un método debe llamarse según el tipo real del objeto.

Este mecanismo es la base del polimorfismo en tiempo de ejecución, ya que permite que una referencia de tipo general invoque métodos que se resuelven dinámicamente según la subclase real. Sin ligadura dinámica, el polimorfismo quedaría muy limitado.

En Java, la ligadura dinámica ocurre automáticamente para métodos de instancia, sin necesidad de indicarlo explícitamente. En C++, en cambio, es necesario declarar los métodos como virtual para habilitar este comportamiento. En Python, este comportamiento es el predeterminado debido a su naturaleza dinámica, ya que los métodos se resuelven en tiempo de ejecución sin necesidad de palabras clave adicionales.


## 3. Pon un ejemplo sencillo en Java, de un `Soldado`, con un método `saluda`, con dos subclases: `Zapador` y `Artillero`, donde `Zapador` sobreescribe el método `saludar`, sustituyendo por completo su comportamiento. Ilustra el funcionamiento del polimorfismo creando un array de `Soldados` de dos tipos y luego recorriéndolo empleando referencias de tipo `Soldado` y llamando a `saludar`.

El polimorfismo permite que un mismo método tenga comportamientos distintos dependiendo del tipo real del objeto. En este caso, Zapador redefine completamente el método saludar, mientras que Artillero puede mantener el comportamiento original de Soldado.

public class Soldado {
    public void saludar() {
        System.out.println("Soy un soldado");
    }
}

public class Zapador extends Soldado {
    @Override
    public void saludar() {
        System.out.println("Soy un zapador especializado");
    }
}

public class Artillero extends Soldado {
    public void saludar() {
        System.out.println("Soy un artillero preparado");
    }
}

public class Main {
    public static void main(String[] args) {

        Soldado[] soldados = {
            new Soldado(),
            new Zapador(),
            new Artillero()
        };

        for (Soldado s : soldados) {
            s.saludar();
        }
    }
}

En el recorrido del array se observa que, aunque la referencia es de tipo Soldado, el método ejecutado depende del tipo real del objeto. Esto demuestra el comportamiento polimórfico mediante ligadura dinámica.


## 4. Si sobreescribo un método, ¿puedo invocar el método base para trabajar a partir de su resultado? Haz que zapador cambie ligeramente la forma de saludar, que salude de forma normal, tal cual hace el soldado base, pero que además añada un "ZAPADOR A SUS ORDENES" ¿qué palabra clave del lenguaje has usado para invocar al método de la clase base?

Cuando se sobreescribe un método, es posible reutilizar el comportamiento de la clase base y ampliarlo en la subclase. Esto permite modificar parcialmente la lógica sin perder la funcionalidad original definida en la superclase.

En Java, para invocar el método de la clase base desde una subclase se utiliza la palabra clave super. Esta palabra permite acceder a la implementación original del método heredado.

public class Soldado {
    public void saludar() {
        System.out.println("Soy un soldado");
    }
}

public class Zapador extends Soldado {

    @Override
    public void saludar() {
        super.saludar();
        System.out.println("ZAPADOR A SUS ORDENES");
    }
}

En este caso, el método saludar del Zapador primero ejecuta el comportamiento del Soldado y luego añade su propio mensaje, demostrando una extensión del comportamiento base.


## 5. Al sobreescribir un método en Java, ¿qué restricciones existen sobre los tipos de los parámetros y el tipo de retorno? ¿Qué diferencia hay entre sobreescritura (*overriding*) y sobrecarga (*overloading*)? ¿Para qué sirve la anotación `@Override` y por qué es recomendable usarla siempre?

Al sobreescribir un método en Java, la firma del método debe coincidir con la de la clase base, es decir, los parámetros deben ser exactamente los mismos en tipo y orden. El tipo de retorno debe ser el mismo o un subtipo covariante del definido en la superclase. No se permite cambiar la lista de parámetros.

La sobreescritura (overriding) ocurre cuando una subclase redefine un método heredado con la misma firma, modificando su comportamiento. En cambio, la sobrecarga (overloading) consiste en definir varios métodos con el mismo nombre pero con diferentes parámetros dentro de la misma clase, sin relación de herencia necesaria.

La anotación @Override sirve para indicar explícitamente que un método está siendo sobreescrito. Aunque no es obligatoria, es recomendable porque permite al compilador detectar errores, como firmas incorrectas o intentos fallidos de sobreescritura, evitando así errores difíciles de depurar en tiempo de ejecución

## 6. Entonces, cuando se estudia Java, ¿se emplea el polimorfismo desde el principio? Por ejemplo, sobreescribiendo `toString` o sobreescribiendo `equals`, ¿ya estoy usando polimorfismo?

Sí, el polimorfismo se utiliza desde etapas muy tempranas en Java, incluso cuando no se presenta de forma explícita como concepto. Cada vez que se sobreescribe un método heredado de Object, como toString() o equals(), se está aplicando polimorfismo mediante sobreescritura.

Esto ocurre porque estos métodos están definidos en la clase base Object, y las subclases proporcionan su propia implementación. Cuando se invoca toString() sobre una referencia de tipo Object, el método ejecutado depende del tipo real del objeto en tiempo de ejecución.

Por tanto, aunque no se esté usando jerarquías complejas, ya se está aprovechando el mecanismo de ligadura dinámica propio del polimorfismo.


## 7. ¿Qué es una **"clase abstracta"**? ¿Qué es un **"método abstracto"**? ¿Puedo crear instancias de una clase abstracta? Pongamos un ejemplo en Java: Redefinamos `Soldado`, hagamos que, además del método `saluda` que ya tenía, tenga un método `atacar`, que sea abstracto y que cada tipo de soldado haga su acción cuando se le pida atacar. ¿Donde debemos poner `abstract`?

Una clase abstracta es una clase que no puede ser instanciada directamente y que se utiliza como base para otras clases. Su objetivo es definir una estructura común que debe ser extendida por subclases, pudiendo incluir tanto métodos implementados como métodos sin implementación.

Un método abstracto es un método que se declara sin cuerpo, es decir, sin implementación, y que obliga a las subclases a proporcionarla. Esto permite definir un contrato de comportamiento que las subclases deben cumplir.

Una clase abstracta no puede instanciarse porque puede contener métodos incompletos, por lo que solo tiene sentido como clase base. La palabra clave abstract se coloca tanto en la declaración de la clase como en los métodos abstractos.

public abstract class Soldado {

    public void saludar() {
        System.out.println("Soy un soldado");
    }

    public abstract void atacar();
}

public class Zapador extends Soldado {

    @Override
    public void atacar() {
        System.out.println("El zapador coloca minas");
    }
}

public class Artillero extends Soldado {

    @Override
    public void atacar() {
        System.out.println("El artillero dispara cohetes");
    }
}




## 8. ¿Qué efecto tiene la palabra clave `final` sobre métodos y clases en Java? ¿Cómo se relaciona con el polimorfismo? ¿Conoces algún ejemplo de clase `final` en la propia API estándar de Java?
La palabra clave final en Java restringe la capacidad de modificación de clases, métodos o variables. Cuando se aplica a una clase, impide que pueda ser heredada. Cuando se aplica a un método, impide que pueda ser sobrescrito por una subclase.

Esto tiene una relación directa con el polimorfismo, ya que limita la posibilidad de sobreescritura y, por tanto, de comportamiento dinámico. Un método final no participa en el polimorfismo de sobreescritura, ya que su implementación queda fija.

Un ejemplo de clase final en la API estándar de Java es String. Esta clase no puede ser extendida, lo que garantiza su comportamiento inmutable y seguro.


## 9. En Java, qué son las **"interfaces"**? ¿Son como clases abstractas? ¿Una clase puede implementar más de una interfaz?

Las interfaces en Java son contratos que definen un conjunto de métodos que una clase debe implementar. No contienen normalmente implementación de estado (aunque pueden tener métodos por defecto en versiones modernas), y su objetivo es definir comportamiento sin imponer una jerarquía de clases.

Se parecen a las clases abstractas en que ambas pueden definir métodos que deben ser implementados por las subclases, pero las interfaces son más flexibles porque no implican herencia de estado y permiten una forma más pura de abstracción.

Una clase en Java puede implementar múltiples interfaces, lo que permite una forma de “herencia múltiple de comportamiento”, evitando los problemas asociados a la herencia múltiple de clases.


## 10. Vamos a poner un ejemplo nuevo con polimorfismo. Queremos implementar una clase `Punto`, con un método `calcularDistanciaA`, que permite calcular la distancia a otro `Punto`. Sin embargo, como queremos trabajar con puntos 2D y 3D, haz que ese método sea abstracto y haya dos implementaciones de ese cálculo de distancia. Emplea `instanceof` y *downcasting* para verificar que se recibe un punto compatible y poder calcular correctamente la distancia siempre entre puntos del mismo subtipo. Aprovecha este diseño para crear ahora una clase `Linea`, que acepta `Punto`, sin saber de qué tipo es, y es capaz de dar su longitud independientemente de las dimensiones de sus puntos (las cuales desconoce).

Se define una clase abstracta Punto que actúa como tipo común para puntos 2D y 3D. El método calcularDistanciaA se declara como abstracto para obligar a cada subclase a implementar su propia lógica de distancia.

public abstract class Punto {
    public abstract double calcularDistanciaA(Punto otro);
}

En los puntos 2D y 3D se utiliza instanceof y downcasting para asegurar que solo se calcula la distancia entre puntos del mismo tipo.

public class Punto2D extends Punto {

    private double x, y;

    public Punto2D(double x, double y) {
        this.x = x;
        this.y = y;
    }

    @Override
    public double calcularDistanciaA(Punto otro) {
        if (!(otro instanceof Punto2D)) {
            throw new IllegalArgumentException("Tipos incompatibles");
        }

        Punto2D p = (Punto2D) otro;

        double dx = p.x - this.x;
        double dy = p.y - this.y;

        return Math.sqrt(dx * dx + dy * dy);
    }
}

public class Punto3D extends Punto {

    private double x, y, z;

    public Punto3D(double x, double y, double z) {
        this.x = x;
        this.y = y;
        this.z = z;
    }

    @Override
    public double calcularDistanciaA(Punto otro) {
        if (!(otro instanceof Punto3D)) {
            throw new IllegalArgumentException("Tipos incompatibles");
        }

        Punto3D p = (Punto3D) otro;

        double dx = p.x - this.x;
        double dy = p.y - this.y;
        double dz = p.z - this.z;

        return Math.sqrt(dx * dx + dy * dy + dz * dz);
    }
}

La clase Linea trabaja únicamente con el tipo abstracto Punto, sin conocer si son 2D o 3D. El polimorfismo permite que la distancia se resuelva correctamente según el tipo real de los puntos.

public class Linea {
    private Punto a;
    private Punto b;

    public Linea(Punto a, Punto b) {
        this.a = a;
        this.b = b;
    }

    public double longitud() {
        return a.calcularDistanciaA(b);
    }
}




## 11. ¿Qué es la **"herencia de interfaces"** en Java? ¿Existe **"herencia múltiple de interfaces"**? Pon un ejemplo de una interfaz `Fichero` que tenga un método para leer su contenido en forma de `String` y luego dicha interfaz sea extendida por otra que sea `FicheroEscribible` que permita enviar contenido e incluso eliminar el fichero.

La herencia de interfaces consiste en que una interfaz puede extender otra interfaz, heredando sus métodos y ampliando su contrato. Esto permite construir jerarquías de especificaciones sin introducir estado ni problemas de herencia de clases.

En Java sí existe herencia múltiple de interfaces, es decir, una interfaz puede extender varias interfaces al mismo tiempo. Esto es posible porque las interfaces no contienen implementación de estado que pueda generar conflictos.

public interface Fichero {
    String leer();
}

public interface Fichero {
    String leer();
}

public interface FicheroEscribible extends Fichero {
    void escribir(String contenido);
    void eliminar();
}

En este ejemplo, FicheroEscribible hereda el método leer() de Fichero y añade nuevas operaciones. Esto permite diseñar sistemas flexibles donde las clases pueden implementar solo las capacidades que necesiten sin quedar atadas a una jerarquía rígida.



# TEMA 6. Genericidad

## 1. Empleando `void*` en C o `Object` en Java, pon un ejemplo de una estructura de datos, que empleando un array primitivo, permita alojar cualquier tipo de dato.

En C puede utilizarse void* para almacenar referencias a cualquier tipo de dato, ya que void* representa un puntero genérico. Esto permite construir estructuras de datos reutilizables, aunque el lenguaje pierde información sobre el tipo real almacenado.

El siguiente ejemplo implementa una estructura sencilla basada en un array de punteros genéricos. Puede almacenar enteros, cadenas u otros tipos, aunque el programador debe recordar manualmente el tipo real de cada elemento.

#include <stdio.h>

struct Lista {
    void* datos[10];
    int numElementos;
};

void add(struct Lista* l, void* dato) {
    l->datos[l->numElementos] = dato;
    l->numElementos++;
}

int main() {

    struct Lista lista;
    lista.numElementos = 0;

    int edad = 20;
    char* nombre = "Ana";

    add(&lista, &edad);
    add(&lista, nombre);

    printf("%d\n", *((int*) lista.datos[0]));
    printf("%s\n", (char*) lista.datos[1]);

    return 0;
}


En Java ocurre algo parecido usando Object, ya que todas las clases heredan de Object. Por ello, una estructura basada en Object[] puede almacenar cualquier objeto.

public class Lista {

    private Object[] datos = new Object[10];
    private int numElementos = 0;

    public void add(Object o) {
        datos[numElementos] = o;
        numElementos++;
    }

    public Object get(int pos) {
        return datos[pos];
    }
}

public class Main {

    public static void main(String[] args) {

        Lista lista = new Lista();

        lista.add("Hola");
        lista.add(25);

        String texto = (String) lista.get(0);
        Integer numero = (Integer) lista.get(1);

        System.out.println(texto);
        System.out.println(numero);
    }
}

## 2. Brevemente, ¿Qué significa la **programación genérica**? ¿Es el ejemplo anterior un ejemplo básico de programación genérica? 

La programación genérica consiste en crear algoritmos y estructuras de datos reutilizables que puedan trabajar con distintos tipos sin necesidad de duplicar código. El objetivo es escribir una única implementación válida para muchos tipos diferentes.

Por ejemplo, una lista dinámica debería poder almacenar String, Integer o cualquier otro tipo sin crear una versión distinta de la lista para cada caso. La genericidad permite precisamente reutilizar la misma estructura de datos manteniendo flexibilidad.

El ejemplo anterior sí puede considerarse un ejemplo básico de programación genérica, porque la estructura permite trabajar con distintos tipos de datos. Sin embargo, se trata de una genericidad poco segura, ya que el compilador no controla los tipos almacenados y obliga a realizar conversiones manuales (cast).

Por este motivo, lenguajes modernos incorporan mecanismos más avanzados como los generics de Java o los templates de C++, que permiten mantener genericidad y seguridad de tipos simultáneamente.

## 3. Indica los problemas respecto al chequeo de tipos, de emplear `void*` o `Object` cuando se crean estructuras de datos genéricas. 

El principal problema de utilizar void* en C o Object en Java es la pérdida de información de tipos. La estructura puede almacenar cualquier dato, pero el compilador deja de saber qué tipo concreto contiene cada posición.

Esto obliga a realizar conversiones manuales (casting) al recuperar los elementos. Si el programador se equivoca en el tipo real almacenado, pueden producirse errores en tiempo de ejecución. En C esto puede causar comportamientos impredecibles o corrupción de memoria, mientras que en Java normalmente aparece una excepción ClassCastException.

Lista lista = new Lista();

lista.add("Hola");

Integer n = (Integer) lista.get(0);

En este ejemplo, el código compila correctamente porque Object acepta cualquier objeto. Sin embargo, en ejecución se intentará convertir un String a Integer, provocando un error.

Además, el código se vuelve menos legible y más difícil de mantener, ya que el tipo real de los datos no queda reflejado claramente en la estructura ni en sus métodos.


## 4. Vamos entonces con mecanismos de mejora de la programación genérica ¿Qué son los **parámetros de tipo**? 

Los parámetros de tipo son mecanismos que permiten definir clases, interfaces o métodos genéricos indicando tipos de manera abstracta. En lugar de trabajar directamente con Object, se utilizan símbolos como T, E o K que representan un tipo concreto que será especificado posteriormente.

Esto permite crear estructuras reutilizables manteniendo seguridad de tipos. El compilador conoce el tipo concreto utilizado y puede detectar errores antes de ejecutar el programa, evitando conversiones inseguras.

public class Caja<T> {

    private T valor;

    public void set(T valor) {
        this.valor = valor;
    }

    public T get() {
        return valor;
    }
}

Caja<String> caja = new Caja<>();

caja.set("Hola");

String texto = caja.get();

En este ejemplo, T actúa como parámetro de tipo. Cuando se instancia Caja<String>, el compilador sustituye T por String, permitiendo recuperar valores sin necesidad de casting explícito.


## 5. En Java existe "generics", en C++ existen "templates". Pon un ejemplo de uso de programación genérica en ambos, instanciando una lista o vector dinámico que solo admite `String`. Introduce valores, y luego haz un recorrido de ellos mostrando cómo cada elemento es del tipo concreto con seguridad.

En Java, los generics permiten parametrizar clases e interfaces indicando qué tipo concreto almacenarán. Esto hace posible que el compilador compruebe los tipos automáticamente y evite conversiones inseguras.

import java.util.ArrayList;
import java.util.List;

public class Main {

    public static void main(String[] args) {

        List<String> textos = new ArrayList<>();

        textos.add("Hola");
        textos.add("Mundo");

        for (String s : textos) {
            System.out.println(s.toUpperCase());
        }
    }
}

En este caso, la lista solo admite objetos String. El compilador impide introducir objetos de otros tipos y cada elemento recuperado ya es directamente un String, sin necesidad de casting.

En C++, los templates proporcionan un mecanismo similar de genericidad. La biblioteca estándar incluye vector, que puede parametrizarse con el tipo deseado.

#include <iostream>
#include <vector>
#include <string>

int main() {

    std::vector<std::string> textos;

    textos.push_back("Hola");
    textos.push_back("Mundo");

    for (std::string s : textos) {
        std::cout << s << std::endl;
    }

    return 0;
}

En ambos lenguajes se consigue reutilización de estructuras de datos y seguridad de tipos al mismo tiempo. El compilador detecta automáticamente usos incorrectos antes de ejecutar el programa, reduciendo muchos errores frecuentes de la programación genérica tradicional basada en Object o void*.



## 6. Sobre el funcionamiento de la programación genérica. ¿Qué hace el compilador cuando se instancia una clase que tiene parámetros de tipo? ¿Hace lo mismo C++ y Java? ¿Qué es el "type erasure" de Java y la "instanciación de plantillas" de C++?

Cuando se utiliza programación genérica, el compilador debe transformar el código genérico en código ejecutable concreto. Para ello, sustituye los parámetros de tipo por tipos reales y comprueba que las operaciones realizadas sean compatibles con dichos tipos.

Sin embargo, Java y C++ implementan este proceso de manera distinta. En Java, los generics utilizan un mecanismo llamado type erasure (borrado de tipos). Esto significa que, durante la compilación, la información genérica se usa para verificar tipos, pero posteriormente se elimina y el código generado trabaja realmente con referencias de tipo Object o con el límite superior indicado.

Por ejemplo:

List<String> textos = new ArrayList<>();

Tras el type erasure, el compilador genera internamente algo equivalente a:

List textos = new ArrayList();

añadiendo automáticamente los cast necesarios donde corresponda.

En C++, en cambio, los templates funcionan mediante instanciación de plantillas. El compilador genera versiones reales y separadas del código para cada tipo utilizado. Por ejemplo, un vector<string> y un vector<int> producen implementaciones distintas en el código compilado.

Esto hace que C++ conserve completamente la información de tipos en tiempo de compilación, mientras que Java sacrifica parte de ella para mantener compatibilidad con versiones antiguas de la máquina virtual y de la API.


## 7. Vamos a crear una nueva clase con parámetros de tipo. Define en Java una clase `Par`, que permite alojar dos valores de tipos diferentes. Incluye un constructor y un getter para cada tipo. Pon un ejemplo de uso de ese `Par`, por ejemplo para especificar el tipo de retorno de una función que devuelve en un `Par` la media y desviación típica de un array de `double`. 

Los parámetros de tipo permiten definir clases reutilizables capaces de trabajar con distintos tipos concretos manteniendo seguridad de tipos. En este caso, se define una clase Par capaz de almacenar dos valores potencialmente diferentes.

public class Par<T1, T2> {

    private T1 primero;
    private T2 segundo;

    public Par(T1 primero, T2 segundo) {
        this.primero = primero;
        this.segundo = segundo;
    }

    public T1 getPrimero() {
        return primero;
    }

    public T2 getSegundo() {
        return segundo;
    }
}

La clase puede utilizarse para representar el retorno de funciones que necesiten devolver varios resultados simultáneamente. Por ejemplo, una función estadística que devuelve media y desviación típica.

public class Estadistica {

    public static Par<Double, Double>
    calcular(double[] datos) {

        double suma = 0;

        for (double d : datos) {
            suma += d;
        }

        double media = suma / datos.length;

        double sumaDif = 0;

        for (double d : datos) {
            double dif = d - media;
            sumaDif += dif * dif;
        }

        double desviacion =
            Math.sqrt(sumaDif / datos.length);

        return new Par<>(media, desviacion);
    }
}
public class Main {

    public static void main(String[] args) {

        double[] datos = {2, 4, 6, 8};

        Par<Double, Double> r =
            Estadistica.calcular(datos);

        System.out.println(
            "Media: " + r.getPrimero()
        );

        System.out.println(
            "Desviación: " + r.getSegundo()
        );
    }
}

Gracias a los generics, el compilador sabe exactamente qué tipo almacena cada posición del Par, evitando conversiones inseguras y mejorando la legibilidad del código.


## 8. En Java, se pueden declarar parámetros de tipo también a nivel de método, no solo a nivel de clase. Pon un ejemplo con un método genérico `seleccionaUno`, que pasados dos objetos del mismo tipo, te devuelva aleatoriamente uno de ellos. Muestra la diferencia de definirlo con dos `Object`, a definirlo con dos parámetros de tipo, en terminos de (i) evitar downcasting y (ii) forzar que ambos objetos sean del mismo tipo. 

En Java, los parámetros de tipo pueden declararse también directamente en métodos. Esto permite crear operaciones genéricas sin necesidad de que toda la clase sea genérica.

Una primera versión basada en Object permite reutilización, pero pierde seguridad de tipos y obliga a realizar downcasting manual.

public static Object seleccionaUno(
        Object a,
        Object b) {

    if (Math.random() < 0.5) {
        return a;
    }

    return b;
}
String s =
    (String) seleccionaUno("Hola", "Adiós");

El compilador no puede garantizar que ambos parámetros sean del mismo tipo ni evitar conversiones inseguras.

Con parámetros de tipo, el método mantiene la información de tipos y evita esos problemas.

public static <T> T seleccionaUno(
        T a,
        T b) {

    if (Math.random() < 0.5) {
        return a;
    }

    return b;
}
String s =
    seleccionaUno("Hola", "Adiós");

En esta segunda versión, el compilador sabe que el retorno es un String, por lo que no se necesita downcasting. Además, se fuerza que ambos parámetros sean compatibles con el mismo tipo T, aumentando la seguridad del programa.


## 9. ¿Se pueden establecer restricciones en los parámetros de tipo? Por ejemplo, si quiero definir un tipo genérico `<T>`, ¿puedo decir que tenga que ser, al menos, un número para poder tratarlo como tal? Pon un ejemplo en Java de un `Punto` con dos coordenadas, metodos `getX`, `getY`, y una función `calcularDistanciaA` otro `Punto`. Permite que esas coordenadas sean cualquier tipo de número. Pon dos soluciones: una simplemente creando coordenadas de tipo `Number` y otra añadiendo generics para reforzar el chequeo de tipos y saber exactamente con qué tipo de número trabaja el `Punto`. En este caso y respecto al "type erasure", ¿cuál es el tipo final tras la compilación?

Sí, Java permite establecer restricciones sobre parámetros de tipo mediante límites (bounds). Esto permite indicar que un tipo genérico debe heredar de una determinada clase o implementar una interfaz concreta.

Por ejemplo, puede indicarse que T debe ser al menos un Number, permitiendo usar operaciones numéricas comunes como doubleValue().

Una primera solución consiste simplemente en utilizar Number directamente.

public class Punto {

    private Number x;
    private Number y;

    public Punto(Number x, Number y) {
        this.x = x;
        this.y = y;
    }

    public Number getX() {
        return x;
    }

    public Number getY() {
        return y;
    }

    public double calcularDistanciaA(Punto p) {

        double dx =
            p.x.doubleValue() - x.doubleValue();

        double dy =
            p.y.doubleValue() - y.doubleValue();

        return Math.sqrt(dx * dx + dy * dy);
    }
}

Esta solución es flexible, pero pierde precisión sobre el tipo exacto utilizado. El compilador no sabe si el punto trabaja con Integer, Double o Float.

Con generics puede reforzarse el chequeo de tipos.

public class Punto<T extends Number> {

    private T x;
    private T y;

    public Punto(T x, T y) {
        this.x = x;
        this.y = y;
    }

    public T getX() {
        return x;
    }

    public T getY() {
        return y;
    }

    public double calcularDistanciaA(
            Punto<T> p) {

        double dx =
            p.x.doubleValue() - x.doubleValue();

        double dy =
            p.y.doubleValue() - y.doubleValue();

        return Math.sqrt(dx * dx + dy * dy);
    }
}
Punto<Double> p1 =
    new Punto<>(1.0, 2.0);

Punto<Double> p2 =
    new Punto<>(4.0, 6.0);

En este caso, el compilador sabe exactamente que el punto trabaja con Double. Sin embargo, debido al type erasure, tras la compilación el tipo genérico T desaparece y queda sustituido por su límite superior, es decir, Number.


## 10. Sobre las soluciones anteriores. Si bien ambas permiten trabajar con distintos tipos de número sin duplicar la clase `Punto`, reflexiona sobre el refuerzo del chequeo de tipos con generics. ¿Permiten ambas crear un punto con una coordenada de tipo entero y la otra coordenada de tipo real? ¿Qué tipo devuelve el `getX` con la solucion sin generics y qué tipo devuelve el que tiene la solución con generics?

Ambas soluciones permiten reutilizar la clase Punto para trabajar con distintos tipos numéricos sin necesidad de duplicar código. Sin embargo, la versión basada únicamente en Number ofrece un chequeo de tipos mucho menos estricto que la versión basada en generics.

En la solución sin generics, las coordenadas son simplemente de tipo Number, por lo que puede construirse un punto mezclando distintos tipos numéricos sin restricciones:

Punto p =
    new Punto(3, 4.5);

Esto compila correctamente porque tanto Integer como Double son subtipos de Number.

En cambio, en la solución genérica:

Punto<Double> p =
    new Punto<>(3.0, 4.5);

el compilador obliga a que ambas coordenadas sean compatibles con el mismo tipo T. Esto refuerza el chequeo de tipos y evita mezclas accidentales de representaciones numéricas distintas.

Además, el método getX() devuelve tipos diferentes en cada diseño. En la solución sin generics, el retorno es simplemente Number:

Number x = p.getX();

por lo que el compilador desconoce el tipo concreto real.

En cambio, con generics el compilador conoce exactamente el tipo utilizado:

Double x = p.getX();

Esto mejora la seguridad de tipos, evita conversiones innecesarias y hace el código más expresivo y fácil de mantener.


## 11. Hagamos un ejemplo avanzado. El siguiente código, con interfaz `Punto`, que define un método `calcularDistanciaA(Punto p)`, junto con las implementaciones `Punto2D` y `Punto3D`. Añade generics para asegurarnos que la sobreescritura del método calcular distancia a otro `Punto` siempre es sobre un `Punto` del mismo tipo, evitando `instanceof` y el downcasting.
```java
public interface Punto { 
    public double distanciaA(Punto p); 
} 

public class Punto2D implements Punto { 
     private final double x, y; 
     public Punto2D(double x, double y) { 
        this.x = x; this.y = y; 
    } 

    @Override 
    public double distanciaA(Punto p) { 
        if (p instanceof Punto2D) { 
            Punto2D p2d = (Punto2D) p; 
            return Math.sqrt(Math.pow(x - p2d.x, 2) 
                    + Math.pow(y - p2d.y, 2)); 
        } else { 
            throw new RuntimeException("p debe ser Punto 2D"); 
        } 
    } 
} 
public class Punto3D implements Punto { 
    // Igual que Punto2D, pero con tres coordenadas
    ...
} 
```

En el diseño original, el método recibe un Punto genérico, por lo que las implementaciones concretas deben comprobar manualmente el tipo mediante instanceof y realizar downcasting. Esto introduce complejidad y desplaza errores potenciales a tiempo de ejecución.

Los generics permiten reforzar el chequeo de tipos haciendo que cada subtipo de punto solo pueda calcular distancias respecto a puntos de su mismo tipo. Así, el compilador detecta automáticamente usos incorrectos y desaparece la necesidad de instanceof.

public interface Punto<T extends Punto<T>> {

    double distanciaA(T p);
}

La interfaz utiliza un parámetro de tipo recursivo (T extends Punto<T>), indicando que el tipo concreto debe ser también un subtipo de Punto.

public class Punto2D
implements Punto<Punto2D> {

    private final double x;
    private final double y;

    public Punto2D(double x, double y) {
        this.x = x;
        this.y = y;
    }

    @Override
    public double distanciaA(Punto2D p) {

        double dx = x - p.x;
        double dy = y - p.y;

        return Math.sqrt(dx * dx + dy * dy);
    }
}
public class Punto3D
implements Punto<Punto3D> {

    private final double x;
    private final double y;
    private final double z;

    public Punto3D(
            double x,
            double y,
            double z) {

        this.x = x;
        this.y = y;
        this.z = z;
    }

    @Override
    public double distanciaA(Punto3D p) {

        double dx = x - p.x;
        double dy = y - p.y;
        double dz = z - p.z;

        return Math.sqrt(
            dx * dx +
            dy * dy +
            dz * dz
        );
    }
}

Ahora el compilador impide automáticamente situaciones incorrectas como intentar calcular la distancia entre un Punto2D y un Punto3D. El chequeo pasa de tiempo de ejecución a tiempo de compilación, aumentando la seguridad del programa.


## 12. Dado que `String` es subtipo de `Object`, ¿significa eso que `List<String>` es subtipo de `List<Object>`? ¿Y que `String[]` es subtipo de `Object[]`? Razona por qué la respuesta es diferente en cada caso y qué problema en tiempo de ejecución puede aparecer con los arrays. A partir de estos ejemplos, define qué significa que un tipo genérico sea **covariante**, **contravariante** o **invariante** respecto a su parámetro de tipo.

Aunque String es subtipo de Object, en Java List<String> no es subtipo de List<Object>. Los tipos genéricos en Java son invariantes por defecto. Esto significa que la relación de herencia entre los parámetros de tipo no se traslada automáticamente al tipo genérico que los contiene.

Por ejemplo, este código es ilegal:

List<String> textos =
    new ArrayList<>();

List<Object> objetos = textos;

Java lo prohíbe porque permitiría insertar objetos que no son String dentro de la lista.

Sin embargo, con arrays ocurre algo distinto: String[] sí es subtipo de Object[]. Los arrays en Java son covariantes.

String[] textos =
    new String[5];

Object[] objetos = textos;

Esto compila correctamente, pero puede provocar errores en ejecución:

objetos[0] = 123;

Aunque objetos parece un Object[], realmente apunta a un String[]. Por ello, en ejecución aparece una excepción ArrayStoreException.

A partir de estos ejemplos:

Un tipo es covariante si conserva la relación de herencia entre tipos (String[] → Object[]).
Un tipo es invariante si no existe relación entre versiones parametrizadas distintas (List<String> no es List<Object>).
Un tipo es contravariante si la relación se invierte, algo que Java permite parcialmente mediante ? super T.


## 13. Java permite recuperar covarianza y contravarianza en tipos genéricos de forma controlada mediante **wildcards**. ¿Qué es un wildcard (`?`)? Muestra la diferencia entre `List<? extends T>` y `List<? super T>`, indicando en qué casos se usa cada uno. Pon dos ejemplos: (i) un método que reciba una lista de números y calcule su suma, usando `? extends`; (ii) un método que reciba una lista y le añada varios números enteros, usando `? super`.

Respuesta

Un wildcard (?) representa un tipo desconocido dentro de un genérico. Permite trabajar con distintos tipos relacionados sin fijar exactamente cuál es el tipo concreto utilizado. Esto proporciona flexibilidad manteniendo seguridad de tipos.

List<? extends T> expresa covarianza controlada. Significa “lista de algún subtipo de T”. Se utiliza principalmente cuando solo interesa leer elementos de la estructura. En cambio, List<? super T> expresa contravarianza controlada y significa “lista de algún supertipo de T”. Se utiliza cuando interesa insertar elementos.

Un ejemplo típico de ? extends es recorrer listas numéricas sin importar el subtipo concreto:

public static double sumar(
        List<? extends Number> nums) {

    double suma = 0;

    for (Number n : nums) {
        suma += n.doubleValue();
    }

    return suma;
}

Este método acepta listas de Integer, Double, Float, etc., porque todas contienen subtipos de Number.

En cambio, ? super resulta útil para insertar valores:

public static void agregarEnteros(
        List<? super Integer> lista) {

    lista.add(10);
    lista.add(20);
    lista.add(30);
}

Este método puede recibir List<Integer>, List<Number> o incluso List<Object>, ya que todos son capaces de almacenar enteros.

Una regla habitual para recordar su uso es:

Producer extends → si la estructura “produce” datos para leer.
Consumer super → si la estructura “consume” datos que se insertan.
