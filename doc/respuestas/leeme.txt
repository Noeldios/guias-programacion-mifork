# TEMA 1. Clases y objetos

## 1. ¿Cuáles son las cuatro características básicas de la programación orientada a objetos? Describe brevemente cada una

Las cuatro características básicas de la programación orientada a objetos son la encapsulación, la abstracción, la herencia y el polimorfismo. Estas permiten organizar el código de forma más estructurada y cercana a cómo se modelan los problemas en el mundo real.

La encapsulación consiste en agrupar datos y funciones en una misma unidad (la clase), controlando además el acceso a esos datos. La abstracción permite representar solo los aspectos relevantes de un objeto, ocultando los detalles innecesarios. Por su parte, la herencia permite crear nuevas clases a partir de otras existentes, reutilizando código. Finalmente, el polimorfismo permite que un mismo método tenga distintos comportamientos según el contexto.


## 2. Cita cuatro lenguajes populares que permitan la programación orientada a objetos

Existen numerosos lenguajes que soportan programación orientada a objetos. Algunos de los más populares son Java, C++, Python y C#. Todos ellos permiten definir clases, crear objetos y aplicar los principios básicos de la POO.

En el caso de C++, se trata de un lenguaje multiparadigma que permite tanto programación estructurada como orientada a objetos. Java, en cambio, está diseñado principalmente bajo el paradigma orientado a objetos. Python y C# también soportan ampliamente este modelo, aunque con diferencias en sintaxis y flexibilidad.


## 3. Los paradigmas anteriores a la POO, ¿Qué es la **programación estructurada**? y, todavía mejor, ¿Qué es la **programación modular**?

La programación estructurada es un paradigma basado en el uso de estructuras de control como secuencias, condicionales y bucles. Su objetivo es mejorar la claridad del código evitando el uso de saltos incontrolados (como goto) y organizando el flujo del programa de forma lógica.

La programación modular va un paso más allá y consiste en dividir un programa en módulos o funciones independientes. Cada módulo realiza una tarea específica y puede reutilizarse en diferentes partes del programa. Esto facilita el mantenimiento, la reutilización y la comprensión del código.

## 4. ¿Qué tres elementos definen a un objeto en programación orientada a objetos?

Un objeto en programación orientada a objetos se define por tres elementos: estado, comportamiento e identidad. Estos conceptos permiten diferenciar unos objetos de otros y describir su funcionamiento.

El estado está representado por los atributos o variables del objeto. El comportamiento corresponde a los métodos o funciones que puede ejecutar. La identidad es lo que permite distinguir un objeto de otro, incluso si tienen los mismos valores en sus atributos.

## 5. ¿Qué es una clase? ¿Es lo mismo que un objeto? ¿Qué es una instancia? ¿Todos los lenguajes orientados a objetos manejan el concepto de clase?

Una clase es una plantilla o modelo que define las propiedades (atributos) y comportamientos (métodos) de un tipo de objeto. No contiene datos concretos, sino la estructura que tendrán los objetos que se creen a partir de ella.

Un objeto no es lo mismo que una clase, sino una instancia de esta. Es decir, una instancia es un objeto concreto creado a partir de una clase. No todos los lenguajes orientados a objetos utilizan clases de la misma forma; algunos, como Java, son completamente basados en clases, mientras que otros como JavaScript utilizan modelos basados en prototipos.


## 6. ¿Dónde se almacenan en memoria los objetos? ¿Es igual en todos los lenguajes? ¿Qué es la **recolección de basura**? 

En Java, los objetos se almacenan normalmente en el heap (memoria dinámica), mientras que las variables que contienen referencias a esos objetos pueden almacenarse en la pila (stack). Este modelo facilita la gestión automática de memoria.

No todos los lenguajes funcionan igual. En C o C++, el programador controla manualmente la memoria, mientras que en Java esta gestión es automática. La recolección de basura es un mecanismo que libera automáticamente la memoria ocupada por objetos que ya no se utilizan, evitando fugas de memoria.


## 7. ¿Qué es un método? ¿Qué es la **sobrecarga de métodos**? 

Un método es una función definida dentro de una clase que describe el comportamiento de los objetos de dicha clase. Es equivalente a una función en C, pero asociada a un tipo concreto de dato.

La sobrecarga de métodos consiste en definir varios métodos con el mismo nombre pero con diferentes parámetros (tipo o número). Esto permite reutilizar el mismo nombre para operaciones similares, mejorando la legibilidad del código.

# TEMA 2. Encapsulación

## 1. En Programación Orientada a Objetos (POO), ¿Qué buscan la **encapsulación** y **la ocultación** de información? Enumera brevemente algunas ventajas de la ocultación de información.

La encapsulación busca agrupar datos y métodos dentro de una clase y controlar cómo se accede a ellos. La ocultación de información consiste en ocultar los detalles internos de la clase y mostrar solo lo necesario.

La principal ventaja es que se reduce la complejidad, se evita el uso incorrecto de los datos y se mejora el mantenimiento del código. También permite cambiar la implementación interna sin afectar al resto del programa.


## 2. ¿Qué se entiende por la **interfaz pública** de un objeto o clase en POO? Describe brevemente cómo se relaciona con la ocultación de información.

La interfaz pública de una clase es el conjunto de métodos y atributos accesibles desde fuera de la clase.

Está formada normalmente por métodos public. La ocultación de información consiste en mantener privados los detalles internos (private) y exponer solo lo necesario mediante la interfaz pública.


## 3. Brevemente: ¿Por qué hay que ser conscientes y diseñar con cuidado la **interfaz pública** de una clase? ¿Es fácil cambiarla?

La interfaz pública es importante porque es cómo se usa la clase.

Es difícil de cambiar porque otros programas pueden depender de ella.

## 4. ¿Qué son las **invariantes de clase** y por qué la ocultación de información nos ayuda?

La interfaz pública es muy importante porque define cómo se usa la clase desde el exterior. Si está mal diseñada, la clase será difícil de usar o poco flexible.

No es fácil cambiarla, ya que otros programas pueden depender de ella. Por eso debe diseñarse con cuidado desde el principio.

## 5. Pon un ejemplo de una clase `Punto` en `Java`, con dos coordenadas, `x` e `y`, de tipo `double`, con un método `calcularDistanciaAOrigen`, y que haga uso de la ocultación de información. ¿Cuál es la interfaz pública de la clase `Punto`? ¿Qué significa `public` y `private`?

class Punto {

        this.y = y;

    public double calcularDistanciaAOrigen() {
        return Math.sqrt(x*x + y*y);
    }

La interfaz pública está formada por el constructor y el método calcularDistanciaAOrigen.

public significa accesible desde fuera de la clase. private significa que solo se puede usar dentro de la propia clase.

## 6. En Java, ¿A quiénes se pueden aplicar los modificadores `public` o `private`?

En Java, public y private se pueden aplicar a atributos, métodos y constructores.

Sirven para controlar la visibilidad de los miembros de una clase.


## 7. En POO, la visibilidad puede ser pública o privada, pero ¿existen más tipos de visibilidad? ¿Qué ocurre en Java? ¿Y en otros lenguajes?

Además de public y private, existe la visibilidad por defecto (package-private) y protected.

En Java se usan estos cuatro niveles. Otros lenguajes pueden tener variantes, pero el concepto general es el mismo: controlar el acceso.


## 8. Responde: Los miembros de instancia privados de un objeto están ocultos para (a) otras clases o (b) otras instancias, aunque sean de la misma clase. Pon un ejemplo añadiendo un método `calcularDistanciaAPunto(Punto otro)` y explica la respuesta.

Los atributos privados no son accesibles desde otras clases, pero sí desde otras instancias de la misma clase.

class Punto {

    public double calcularDistanciaAPunto(Punto otro) {

Esto es posible porque una instancia de la misma clase puede acceder a los miembros privados de otra instancia de esa clase.


## 9. ¿Qué son los métodos "getter" y "setter" en los lenguajes orientados a objetos?

Los getters son métodos que devuelven el valor de un atributo privado.

Los setters son métodos que modifican el valor de un atributo privado de forma controlada.


## 10. Cuando nos referimos a que la ocultación de información mejora la "seguridad" del programa, ¿nos referimos a que no pueda ser "hackeado"?

La “seguridad” no significa protección contra hackers.

Significa evitar errores de programación, como modificar datos de forma incorrecta o romper reglas internas del objeto.

## 11. ¿Qué diferencia hay entre **miembro de instancia** y **miembro de clase**? ¿Los miembros de clase también se pueden ocultar?
Un miembro de instancia pertenece a cada objeto individual. Un miembro de clase (static) pertenece a la clase en general.

Los miembros de clase también pueden ser private o public, igual que los de instancia.


## 12. Brevemente: ¿Tiene sentido que los constructores sean privados?
Sí, puede tener sentido. Se usan para controlar la creación de objetos.

## 13. ¿Cómo se indican los **miembros de clase** en Java? Pon un ejemplo, en la clase `Punto` definida anteriormente, para que incluya miembros de clase que permitan saber cuáles son los valores `x` e `y` máximos que se han establecido en todos los puntos que se hayan creado hasta el momento.

Los miembros de clase se indican con static.

class Punto {
    private double x, y;

    private static double maxX = Double.NEGATIVE_INFINITY;
    private static double maxY = Double.NEGATIVE_INFINITY;

        this.x = x;
        this.y = y;

        if (x > maxX) maxX = x;
        if (y > maxY) maxY = y;
    }
}

# TEMA 3. Excepciones

## 1. Empecemos un tema sobre control de errores en lenguajes de programación, con algo básico. En C, donde no existen las excepciones, pongamos un ejemplo de una raíz que toma número flotante positivo. Queremos controlar el error si la función recibe un número negativo. El usuario debe ser informado pero desde fuera de la función `raiz` ¿Cómo indicamos ese error?. Enumera dos opciones diferentes de diseñar, poniendo un ejemplo de código de cada una.

En C no existen excepciones, por lo que el error se controla con valores de retorno o variables de estado.
double raiz(double x) {
    if (x < 0) return -1;
    return sqrt(x);
}

El llamador comprueba el valor devuelto.

double raiz(double x, int *error) {
    if (x < 0) {
        *error = 1;
        return 0;
    }
    *error = 0;
    return sqrt(x);
}

El error se comunica fuera de la función mediante un parámetro.


## 2. Brevemente ¿Qué es una **"excepción"**? ¿Con qué objetivo las usa un programador cuando implementa funciones o cuando las llama?

Una excepción es un mecanismo para indicar errores en tiempo de ejecución.

Permite separar el código normal del código de error. Se usa para propagar errores automáticamente hasta que alguien los controle.


## 3. Reescribe el mismo ejemplo de raiz, pero en Java, metiendo ese método en una clase `Calculadora` y llama a dicho método desde el método `main`, mostrando cómo se puede controlar desde fuera.

class Calculadora {
    static double raiz(double x) {
        if (x < 0) throw new IllegalArgumentException("Negativo");
        return Math.sqrt(x);
    }
}

public class Main {
    public static void main(String[] args) {
        try {
            System.out.println(Calculadora.raiz(-5));
        } catch (IllegalArgumentException e) {
            System.out.println("Error: " + e.getMessage());
        }
    }
}


## 4. ¿Qué es **"lanzar"** una excepción? ¿Qué es **"controlar"** o **"capturar"** una excepción? ¿Qué es que se **"propague"** una excepción? ¿Qué le va ocurriendo a las funciones en la pila de llamadas por donde se va propagando la excepción? ¿Las funciones que no la controlan se reanudan después de alguna forma? Explica con el mismo ejemplo anterior en Java de la raíz cuadrada.

Lanzar (throw) = crear una excepción cuando ocurre un error.
Capturar (catch) = manejar la excepción cuando ocurre.
Propagar = dejar que la excepción suba por la pila de llamadas.

class Calculadora {
    static double raiz(double x) {
        if (x < 0) throw new IllegalArgumentException();
        return Math.sqrt(x);
    }
}

Si no se captura, la excepción sube por las funciones llamadas hasta main.

Las funciones intermedias que no la capturan no continúan después del error, se aborta su ejecución.

## 5. ¿Qué ventajas tiene frente a C, la **"propagación natural"** de las excepciones a través de la pila (*stack*) de llamadas?

La propagación automática evita comprobar errores en cada nivel.

El código es más limpio porque no se mezclan resultados con códigos de error. Además, se centraliza el manejo en un solo punto.

## 9. Si las excepciones producen rupturas en el código llamador, ¿cómo podemos garantizar que se ejecuta siempre finalmente un código necesario para cierre de ficheros, liberacion de recursos, antes de que continúe propagándose la excepción? Pon un ejemplo en Java con `finally`, tanto con `catch` como sin él.

finally se usa para ejecutar código siempre, haya error o no.
try {
    System.out.println(Calculadora.raiz(-5));
} catch (Exception e) {
    System.out.println("Error");
} finally {
    System.out.println("Siempre se ejecuta");
}

También puede existir sin catch:

try {
    System.out.println("Proceso");
} finally {
    System.out.println("Cierre");
}

## 11. En Java, qué son las excepciones **"controladas"** y las **"no controladas"**? ¿Qué papel juega `RuntimeException`? ## 16. ¿Tiene sentido lanzar excepciones dentro del `catch`? ¿Se puede relanzar la misma excepción capturada? ¿Cuándo tendría sentido hacer esto último? Pon ejemplos de ambos casos.

### RespuestaSí, se puede lanzar dentro de un catch.

También se puede relanzar la misma excepción.

try {
    int x = 5 / 0;
} catch (ArithmeticException e) {
    throw new RuntimeException("Error matemático", e);
}

Se hace cuando se quiere añadir contexto o transformar el error.

## 17. ¿En qué consiste que una excepción sea la **"causa"** de otra excepción? Pon un ejemplo en Java, donde capturemos una excepción de bajo nivel y la encapsulemos en otra personalizada de alto nivel. Cuando una excepción sale por pantalla y tiene una causa, ¿se ve?

Una excepción puede contener otra como causa, formando una cadena de errores.

try {
    Integer.parseInt("a");

    throw new RuntimeException("Error de conversión", e);
}

Sí, la causa se muestra en el stack trace, permitiendo ver el origen real del problema.
